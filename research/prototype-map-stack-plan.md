## 리스크와 대응

### 리스크 1. SQLite 성능 한계

SQLite는 프로토타입에는 적합하지만, 데이터가 커지고 동시 사용자가 많아지면 한계가 분명합니다.

대응:

- 프로토타입은 소규모 데이터셋 기준으로 설계
- 대용량/동시성은 차기 단계에서 PostGIS 전환 전제로 정리

### 리스크 2. GeoJSON 과다 사용

GeoJSON은 구현이 쉽지만, 데이터가 많아지면 브라우저 렌더링이 급격히 느려질 수 있습니다.

대응:

- 샘플 수 제한
- 지도 줌레벨에 따라 데이터 축소
- 운영 전환 시 vector tile 구조 검토

### 리스크 3. Python 템플릿 기반 UI의 복잡도 증가

초기에는 빠르지만, 필터/패널/시계열 상호작용이 커지면 프런트 코드가 복잡해질 수 있습니다.

대응:

- 프로토타입에서는 최소 상호작용만 우선 구현
- 향후 필요 시 React 기반 프런트 분리 검토

### 리스크 4. 공간 연산 부족

SQLite 단독은 PostGIS처럼 강력한 공간 질의를 제공하지 않습니다.

대응:

- 프로토타입에서는 bbox 필터, 단순 좌표 조회 위주로 제한
- 복잡한 공간 분석 요구는 후속 단계에서 재설계

## 개발 방향

본 프로토타입은 OpenStreetMap 기반 지도를 활용하여 3D 스타일의 시각적 표현과 부가정보 오버레이 방식을 검증하는 웹서비스로 개발한다. 프런트엔드는 Python 웹프레임워크 기반 구조 위에 MapLibre GL JS와 deck.gl을 적용하여 지도와 데이터 오버레이를 구현하고, 백엔드는 SQLite 기반의 경량 데이터 저장소와 API 서버로 구성한다. 데이터 제공은 프로토타입 특성상 GeoJSON 및 경량 API 중심으로 단순화하며, 우측 상세패널, hover tooltip, 필터, 시계열 토글 등 핵심 사용자 경험 요소를 우선 구현한다. 본 단계에서는 운영형 대용량 구조보다 기능 검증과 사용자 경험 검증을 우선하며, 향후 필요 시 PostGIS 및 vector tile 기반 구조로 확장 가능하도록 설계한다.

---

## 최종 권장안

프로토타입은 SQLite + GeoJSON + FastAPI + MapLibre + deck.gl로 단순화하고, 운영 전환 시 PostGIS와 vector tile로 확장한다.
