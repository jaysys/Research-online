# 온디바이스 VLA 지능 개발 사전지식 조사자료

- 작성일: 2026-03-15
- 작성 범위: 외부 통신 없이 로봇이 스스로 판단하고 행동하는 `온디바이스 VLA(Vision-Language-Action)` 지능 개발을 위한 기술 배경, 설계 쟁점, 구현 로드맵
- 작성 원칙:
  - 현재 작업 환경에서 외부 검색 없이 정리했다.
  - 특정 수치·제품·논문 버전처럼 변동 가능한 정보는 `일반적 경향` 중심으로 정리했다.
  - 바로 설계와 구현 판단에 쓰일 수 있도록 `개념`보다 `시스템 관점`을 우선했다.

## 1. 한 줄 요약

- 온디바이스 VLA 지능은 `카메라/센서 입력`과 `언어 지시`, `로봇 상태`를 함께 받아 `다음 행동`을 직접 또는 계층적으로 산출하는 로봇용 멀티모달 정책이다.
- 핵심 난점은 모델 정확도 자체보다 `실시간성`, `안전성`, `데이터 품질`, `행동 표현(action representation)`, `장기 과업 분해`, `로봇 하드웨어 제약`의 균형이다.
- 실무적으로는 `거대 단일 모델 1개`보다 `경량 VLA + 상태추정 + 안전제어 + 로컬 메모리 + 태스크 관리자`의 모듈형 구성이 더 현실적이다.

## 2. VLA란 무엇인가

VLA는 보통 다음 세 요소를 하나의 정책 계열 안에서 연결한다.

- `Vision`: RGB, depth, wrist camera, fisheye, event camera, tactile proxy, object mask
- `Language`: 자연어 목표, 제약조건, 작업 순서, 질의응답
- `Action`: 로봇 팔 제어, 모바일 베이스 제어, 그리퍼 제어, 고수준 스킬 호출

전통적인 로봇 스택은 보통 아래처럼 분리된다.

- 인식 모듈
- 상태 추정 모듈
- 경로 계획 모듈
- 제어 모듈
- 태스크 로직

VLA는 이 경계를 일부 줄여서 `관측 -> 의미 해석 -> 행동 결정`을 더 직접 연결한다. 다만 실제 로봇에서는 저수준 제어까지 전부 엔드투엔드로 묶기보다, `고수준 행동 결정은 VLA`, `저수준 안정화는 기존 제어기`로 나누는 편이 안전하다.

## 3. 왜 온디바이스여야 하는가

외부 통신 없이 동작하는 요구는 단순한 오프라인 모드가 아니라 시스템 구조를 크게 바꾼다.

- 지연시간 감소: 네트워크 왕복 없이 즉시 추론 가능
- 연결 불안정 대응: 실내 공장, 재난 현장, 지하, 군집 환경에서 유리
- 보안/프라이버시: 영상과 조작 정보가 외부로 나가지 않음
- 안전성: 통신 장애가 바로 행동 마비로 이어지지 않음
- 비용: 지속적인 클라우드 추론 비용과 업링크 비용 절감

대신 아래 비용을 내부에서 감당해야 한다.

- 제한된 연산량과 전력
- 모델 경량화 필요
- 로컬 메모리와 저장소 설계 필요
- 현장 업데이트와 롤백 체계 필요

## 4. 온디바이스 VLA의 권장 시스템 구조

실무적으로는 아래 6계층 구조가 가장 설명력이 높다.

### 4.1 센서 계층

- RGB 카메라
- depth 또는 stereo
- IMU
- joint encoder
- force/torque 또는 접촉 센서
- base odometry

핵심은 `관측을 많이 넣는 것`보다 `시간 동기화`와 `센서 신뢰도 추정`이다.

### 4.2 상태 추정 계층

- robot state vector 생성
- object-centric state 추정
- pose tracking
- free space / occupancy 추정
- 실패 징후 감지

VLA 입력에 원시 이미지와 함께 `정제된 상태 표현`을 같이 넣으면 데이터 효율이 훨씬 좋아진다.

### 4.3 의미 추론 계층

- 언어 지시 파싱
- 현재 상황 설명
- task progress 추론
- affordance 추론
- 불확실성 추정

여기서는 대형 범용 멀티모달 모델의 능력이 매력적이지만, 온디바이스에서는 작고 빠른 모델로 줄여야 한다.

### 4.4 행동 정책 계층

대표 선택지는 세 가지다.

1. `고수준 스킬 선택형`
- 예: `pick`, `place`, `open`, `navigate_to`, `retry_grasp`
- 장점: 안전성과 디버깅이 좋다.
- 단점: 스킬 라이브러리 품질에 크게 의존한다.

2. `저수준 연속 제어형`
- 예: end-effector delta pose, gripper open/close, base velocity
- 장점: 유연성이 높다.
- 단점: 데이터량과 안정화 비용이 크다.

3. `계층형 혼합`
- VLA는 고수준 의도와 서브골을 내고, 하위 스킬/제어기가 실제 궤적을 만든다.
- 실제 제품화에는 이 방식이 가장 현실적이다.

### 4.5 안전 계층

- 충돌 검사
- joint/velocity limit
- forbidden zone
- human proximity rule
- emergency stop
- low-confidence fallback

안전 계층은 VLA 출력보다 우선순위가 높아야 한다.  
즉 `모델이 하라고 해도 안전 계층이 막을 수 있어야` 한다.

### 4.6 로컬 메모리 및 태스크 관리자

- 단기 메모리: 최근 관측, 최근 행동, 실패 이력
- 장기 메모리: 환경 지도, 자주 쓰는 물체 위치, 사용자 선호
- 태스크 관리자: 단계 분해, 재시도 정책, 중단/복구

온디바이스 환경에서는 클라우드 에이전트처럼 긴 컨텍스트를 항상 쌓기 어렵기 때문에, `메모리 구조화`가 매우 중요하다.

## 5. 액션 표현이 가장 중요하다

VLA 설계에서 가장 큰 성능 차이를 만드는 요소 중 하나가 `무엇을 행동 단위로 예측할 것인가`다.

대표 후보는 아래와 같다.

- 토큰화된 스킬 ID
- waypoint 시퀀스
- end-effector pose delta
- joint target
- base velocity command
- gripper state
- stop / ask / retry / recover 같은 메타 액션

실무 판단은 보통 아래가 안전하다.

- 조작 로봇: `고수준 스킬 + 상대 pose delta + gripper action`
- 이동 로봇: `서브골 위치 + 속도 명령`
- 복합 로봇: `태스크 토큰 + 모션 파라미터`

완전 저수준 joint-space 정책은 멋져 보이지만, 하드웨어별 일반화와 안전성에서 비용이 크다.

## 6. 데이터가 모델보다 더 중요하다

온디바이스 VLA는 파운데이션 모델 크기로 승부하기 어렵다. 대신 데이터 설계가 훨씬 중요하다.

### 6.1 필요한 데이터 종류

- 원격조작 시연 데이터
- 성공/실패 로그
- 자기 탐색 데이터
- 시뮬레이션 롤아웃
- 작업 지시 텍스트
- 센서-행동 동기화 시퀀스

### 6.2 데이터 품질 핵심

- 타임스탬프 정합
- 행동 라벨 지연 보정
- 성공 조건 명확화
- 실패 유형 라벨링
- 카메라 위치와 캘리브레이션 일관성
- 작업 문장의 표준화

### 6.3 실무적으로 꼭 필요한 실패 데이터

성공 시연만 모으면 현장성이 급격히 떨어진다. 반드시 아래 로그가 필요하다.

- grasp miss
- slip
- occlusion
- wrong object selected
- blocked path
- door/drawer partial opening failure
- localization drift
- prompt misunderstanding

## 7. 학습 전략

온디바이스 제약을 고려하면 보통 아래 조합이 유효하다.

### 7.1 사전학습

- 멀티모달 인코더 사전학습
- 비디오-언어 정렬
- 행동 없는 대규모 시각 데이터 활용

### 7.2 행동학습

- behavior cloning
- sequence modeling
- diffusion 또는 autoregressive action prediction
- offline RL 또는 preference-based refinement

### 7.3 현장 적응

- 소량 task-specific fine-tuning
- parameter-efficient tuning
- distillation
- test-time adaptation

온디바이스 운영에서는 `학습` 자체보다 `배포 가능한 작은 모델로 얼마나 잘 증류하느냐`가 더 중요하다.

## 8. 실시간성 목표를 먼저 정해야 한다

모델 구조를 정하기 전에 제어 루프 예산을 정해야 한다.

대표적으로 아래 두 루프를 분리한다.

- `고수준 의사결정 루프`: 대략 수 Hz 수준
- `저수준 제어 루프`: 수십~수백 Hz 수준

권장 원칙은 아래와 같다.

- VLA는 모든 주기에 직접 관여하지 않는다.
- 저수준 안정화는 기존 controller가 맡는다.
- VLA 추론 지연이 커질수록 robot watchdog이 개입해야 한다.

즉, `VLA가 늦어도 로봇은 안전하게 유지`되어야 한다.

## 9. 모델 경량화 전략

온디바이스에서는 아래 전략이 거의 필수다.

- 작은 vision backbone 사용
- language module 축소 또는 증류
- action head 분리
- 4bit/8bit 양자화
- KV cache 최소화
- image token 축소
- ROI 중심 인코딩
- sparse attention 또는 짧은 시간창

추론 효율을 위해 자주 쓰는 구조는 아래와 같다.

- `이미지 인코더`는 고정
- `언어 인코더`는 소형
- `fusion block`은 얕게
- `action head`는 태스크별 분기

## 10. 월드모델과 메모리

온디바이스 VLA가 반복 작업과 장기 과업에서 약해지는 이유는 `지금 보는 장면`에는 강하지만 `상태 유지`에는 약하기 때문이다.

그래서 아래 두 층이 중요하다.

- `World State`
  - 방 구조
  - 객체 위치
  - 열림/닫힘 상태
  - 점유 여부
  - 최근 관측 신뢰도

- `Task Memory`
  - 현재 목표
  - 완료 단계
  - 실패 횟수
  - 대안 계획
  - 사람의 최근 지시

이 정보는 전부 LLM/VLA 컨텍스트 창에 넣기보다, 구조화된 상태 저장소로 관리하는 편이 낫다.

## 11. 오프라인 환경에서의 자율성 설계

외부 통신이 없으면 `모르는 것을 물어볼 수 없다`. 따라서 사전에 아래를 내장해야 한다.

- 작업 가능한 스킬 목록
- 물체 카테고리 사전
- 환경 지도 또는 빠른 재구성 기능
- 에러 코드 체계
- 자기 점검 루틴
- 로컬 도움말/운용 규칙

중요한 설계 원칙은 `질문형 지능`보다 `복구형 지능`이다.

즉, 외부 도움 없이도 아래 행동을 할 수 있어야 한다.

- 다시 보기
- 자세 바꾸기
- 다른 grasp point 시도
- 장애물 우회
- 작업 중단 후 안전 위치 복귀
- 사용자에게 로컬 UI로 상태 표시

## 12. 안전성과 실패 복구

VLA 로봇은 높은 확률로 아래 종류의 실패를 겪는다.

- 인식 실패
- 물체 혼동
- 접촉 모델 오차
- 제어 지연
- 환경 변화
- 언어 모호성
- 장기 과업 중 상태 망각

따라서 정책 출력만으로 완결시키지 말고 별도 recovery FSM을 두는 것이 좋다.

예시 흐름:

1. 명령 이해
2. 환경 확인
3. 행동 시도
4. 성공 판정
5. 실패 시 원인 분류
6. 허용된 재시도 수행
7. 반복 실패 시 안전 중단

## 13. 시뮬레이션과 실기 간 격차

온디바이스 VLA는 실제 로봇 데이터가 비싸기 때문에 시뮬레이션을 많이 쓰게 된다.  
하지만 sim-to-real 격차를 줄이지 않으면 현장 성능이 급격히 떨어진다.

핵심 대응은 아래와 같다.

- domain randomization
- sensor noise injection
- lighting variation
- texture variation
- latency injection
- action execution noise
- camera calibration perturbation

또한 실제 데이터와 시뮬레이션 데이터를 섞을 때는 `실기 데이터의 우선순위`를 높게 두는 편이 낫다.

## 14. 평가 지표

단순 성공률만 보면 배치 데모에 속기 쉽다. 최소한 아래 지표가 필요하다.

- task success rate
- first-try success rate
- time-to-completion
- intervention rate
- collision / safety violation count
- recovery success rate
- unseen environment generalization
- unseen object generalization
- power / thermal budget
- on-device latency

특히 온디바이스 조건에서는 `성능 대비 전력`과 `열 안정성`이 매우 중요하다.

## 15. 권장 개발 단계

### 1단계: 좁은 과업 정의

- 예: 테이블 위 특정 물체 집기, 서랍 열기, 지정 위치로 운반
- 환경과 물체 분포를 제한한다.
- 성공 판정을 명확히 정의한다.

### 2단계: 하이브리드 스택 구축

- perception
- state estimation
- skill library
- safety controller
- 작은 VLA policy

처음부터 완전 엔드투엔드보다 이 구성이 현실적이다.

### 3단계: 로그와 재현성 확보

- raw sensor log
- synchronized action log
- episode replay
- failure tagging

로봇 프로젝트는 재현성 없는 데이터가 누적되면 빠르게 막힌다.

### 4단계: 경량화와 배포

- pruning
- quantization
- distillation
- runtime profiling

### 5단계: 장기 과업과 메모리 확장

- 멀티스텝 작업
- 중간 상태 저장
- 복구 정책
- 로컬 플래너 강화

## 16. 추천 구현 원칙

- `VLA가 전부를 결정하게 하지 말 것`
- `안전 제어와 행동 정책을 분리할 것`
- `고수준 추론과 저수준 제어 주기를 분리할 것`
- `실패 로그를 성공 로그만큼 중요하게 다룰 것`
- `로컬 메모리를 구조화할 것`
- `행동 표현을 하드웨어 독립적으로 설계할 것`
- `처음 목표는 범용성보다 반복 가능한 좁은 성공`으로 잡을 것

## 17. 개발 시 자주 하는 오해

### 오해 1. 큰 멀티모달 모델을 올리면 로봇이 바로 똑똑해진다

- 실제 병목은 추론 지능보다 `행동 데이터`, `실시간성`, `복구 로직`, `센서 정합`에 더 자주 있다.

### 오해 2. 엔드투엔드가 항상 더 좋다

- 실제 제품은 계층형 구조가 디버깅, 안전, 유지보수에서 유리하다.

### 오해 3. 언어 이해가 되면 조작도 된다

- 언어 이해와 물리 조작은 전혀 다른 난도다.
- grasp 안정성, 접촉 추정, 지연 대응은 별도 문제다.

### 오해 4. 클라우드 없이도 클라우드 수준 범용성을 쉽게 얻을 수 있다

- 온디바이스는 모델 크기, 메모리, 전력, 냉각, 저장소, 업데이트 경로가 모두 제한된다.
- 따라서 `범용성 최대화`보다 `현장 임무 달성률 최대화`가 더 현실적이다.

## 18. 실무용 기술 체크리스트

- 센서 시간 동기화가 되어 있는가
- 행동 명령의 기준 좌표계가 일관적인가
- 카메라 캘리브레이션 자동 점검이 가능한가
- 실패와 재시도 정책이 정의되어 있는가
- 추론 지연 시 안전 정지 정책이 있는가
- 로컬 로그 저장과 재생 기능이 있는가
- 모델 업데이트 실패 시 롤백이 가능한가
- 열 스로틀링 상황에서 성능 저하를 측정했는가
- 네트워크 없이 초기화부터 임무 수행까지 가능한가

## 19. 결론

- 외부 통신 없이 동작하는 온디바이스 VLA 지능은 `작은 범용 모델 하나`의 문제가 아니라 `로봇 시스템 통합` 문제에 가깝다.
- 성공 확률이 높은 접근은 `경량 VLA + 구조화된 상태 추정 + 스킬 기반 실행기 + 안전 계층 + 로컬 메모리` 조합이다.
- 초기 개발 목표는 `좁은 과업에서 안정적으로 동작하는 반복 가능 시스템`이어야 하며, 그 위에 점진적으로 과업 폭과 자율성을 넓혀야 한다.
- 제품화 관점에서 가장 중요한 질문은 “얼마나 똑똑한가”보다 “통신 없이도 안전하게, 재현 가능하게, 반복 수행 가능한가”다.

## 20. 후속 조사 권장 주제

외부 자료 조사를 이어서 할 경우 아래 주제를 우선 확인하면 좋다.

- 공개 로봇 행동 데이터셋 비교
- 오픈소스 VLA / VLN / manipulation policy 프레임워크
- 로봇용 온디바이스 추론 런타임
- 소형 GPU/NPU/엣지 가속기별 배포 전략
- diffusion policy와 autoregressive policy의 실기 차이
- 모바일 매니퓰레이터 기준 안전 아키텍처 사례
- 로컬 메모리와 태스크 플래너 설계 패턴
