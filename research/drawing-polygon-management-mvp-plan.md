# 도면 기반 영역 선택 및 폴리곤 관리 웹서비스

## MVP 개발계획서

## 1. 개요

본 MVP는 웹에 표출된 도면 또는 지도 화면에서 사용자가 특정 영역을 직접 선택하거나 그려서 폴리곤 형태로 저장하고, 이를 DB에서 관리할 수 있는 기능을 구현하는 것을 목표로 한다.

이번 단계의 목적은 고도화된 GIS 분석 시스템 구축이 아니라 아래 핵심 흐름의 유효성을 빠르게 검증하는 데 있다.

- 도면 또는 지도 화면 표출
- 사용자 영역 선택 또는 폴리곤 작성
- 선택 결과의 좌표 데이터 저장
- 저장된 폴리곤 목록 조회
- 폴리곤 수정, 삭제, 관리

프로토타입 단계이므로 DB는 SQLite를 사용하고, 공간 처리 복잡도는 최소화하여 MVP 기능 검증에 집중한다.

---

## 2. MVP 목표

이번 MVP의 핵심 목표는 아래 5가지다.

1. 웹 화면에 도면 또는 지도 이미지를 표출한다.
2. 사용자가 화면에서 영역을 선택하거나 폴리곤을 직접 그릴 수 있도록 한다.
3. 선택된 영역의 폴리곤 좌표를 DB에 저장한다.
4. 저장된 폴리곤을 목록 및 화면에서 다시 조회할 수 있게 한다.
5. 저장된 폴리곤에 대해 수정, 삭제, 이름 및 속성 관리 기능을 제공한다.

즉, 이번 MVP의 중심은 시각화 자체가 아니라 영역 입력과 저장, 관리 흐름의 검증이다.

## 3. 기능 범위

### 3.1 도면 또는 지도 표출

- 웹 화면에 도면, 배경 이미지, 또는 지도 표시
- 확대, 축소, 이동 기능 제공
- 기준 배경 위에서 영역 작업 가능

### 3.2 영역 선택 및 폴리곤 작성

- 사용자가 마우스로 다각형 영역을 지정
- 클릭 포인트를 순차적으로 입력하여 polygon 생성
- 작성 완료 후 닫힌 폴리곤으로 확정
- 필요 시 사각형 선택 기능 추가 가능

### 3.3 폴리곤 저장

- 폴리곤 좌표 배열 저장
- 이름, 설명, 생성자, 생성일 등 메타정보 저장
- 특정 도면 또는 레이어와 연결 저장

### 3.4 폴리곤 조회

- 저장된 폴리곤 목록 조회
- 화면 재로딩 시 폴리곤 재표출
- 특정 폴리곤 선택 시 상세정보 표시

### 3.5 폴리곤 관리

- 이름 수정
- 속성 수정
- 좌표 수정
- 삭제
- 상태 관리

## 4. 시스템 구성

### 4.1 아키텍처

```text
사용자 브라우저
    │
    │ polygon draw / edit / select
    ▼
Frontend
(FastAPI 정적 서빙 + MapLibre GL JS + draw/edit UI)
    │
    │ REST API
    ▼
Backend API Server
(FastAPI)
    │
    ▼
SQLite
```

### 4.2 구성 요소

#### Frontend

- 도면 또는 지도 렌더링
- 사용자의 폴리곤 입력 처리
- 저장된 폴리곤 시각화
- 선택, 수정, 삭제 UI 제공

#### Backend

- 폴리곤 CRUD API 제공
- 도면별 폴리곤 조회 API 제공
- 좌표 JSON 저장 및 조회 처리

#### Database

- SQLite 사용
- polygon geometry는 MVP 단계에서 GeoJSON 또는 coordinate JSON 문자열로 저장

## 5. 기술 방향

### Frontend

- FastAPI 기반의 간단한 정적 프런트 제공
- MapLibre GL JS로 화면 렌더링 및 레이어 표시
- draw/edit 컨트롤 계층으로 사용자 영역 작성 기능 구현

MapLibre만으로는 드로잉 기능이 부족하므로 draw 컨트롤 계층을 반드시 포함하는 방향이 적절하다.

### Backend

- FastAPI
- 폴리곤 CRUD API
- 도면 단위 조회 API

### DB

- SQLite
- polygon 좌표는 우선 JSON으로 저장
- 복잡한 공간 질의는 MVP 범위에서 제외

### PostGIS 확장 방향

- 운영 단계에서 PostgreSQL + PostGIS 조합으로 전환 가능하도록 API 입력은 GeoJSON 기준 유지
- `polygon_json`은 MVP 저장 포맷으로 유지하고, 전환 시 `geometry(Polygon, 4326)` 컬럼으로 이관
- 영역 포함, 교차, 버퍼, 거리 계산 같은 공간 연산이 필요해지는 시점에 도입
- 다수 사용자의 동시 편집, 대량 polygon 검색, 공간 인덱스 최적화가 필요할 때 우선 검토

## 6. 데이터 모델

MVP에서는 단순한 구조를 우선한다.

### 6.1 drawings

도면 또는 배경 화면 정보

| 컬럼 | 설명 |
| --- | --- |
| id | 도면 ID |
| name | 도면명 |
| image_url | 도면 이미지 또는 리소스 경로 |
| width | 원본 폭 |
| height | 원본 높이 |
| created_at | 생성일 |

### 6.2 polygons

영역 정보 저장

| 컬럼 | 설명 |
| --- | --- |
| id | 폴리곤 ID |
| drawing_id | 소속 도면 ID |
| name | 영역명 |
| description | 설명 |
| polygon_json | 좌표 JSON 또는 GeoJSON |
| color | 표시 색상 |
| status | 상태 |
| created_at | 생성일 |
| updated_at | 수정일 |

### 6.3 polygon_properties

추가 속성이 필요할 경우

| 컬럼 | 설명 |
| --- | --- |
| id | 속성 ID |
| polygon_id | 폴리곤 ID |
| property_key | 속성명 |
| property_value | 속성값 |

초기 MVP에서는 `polygon_properties` 없이 `metadata_json` 컬럼 하나로 통합해도 충분하다.

## 7. 좌표 저장 방식

MVP에서는 SQLite를 사용하므로 PostGIS geometry 타입보다 GeoJSON 저장 방식이 현실적이다.

### 권장 방식

`polygon_json` 컬럼에 GeoJSON 형태 저장

```json
{
  "type": "Polygon",
  "coordinates": [
    [
      [127.001, 37.501],
      [127.002, 37.501],
      [127.002, 37.502],
      [127.001, 37.502],
      [127.001, 37.501]
    ]
  ]
}
```

장점은 아래와 같다.

- 프런트와 주고받기 쉽다
- MapLibre에서 바로 재사용 가능하다
- 추후 PostGIS로 옮기기 쉽다

즉, MVP에서는 좌표를 GeoJSON으로 저장하는 것이 가장 단순하고 확장성도 좋다.

### PostGIS 전환 시 저장 예시

운영 전환 시에는 GeoJSON 입력을 그대로 받아 DB 내부에서 geometry로 변환하는 방식이 자연스럽다.

예시 개념:

- API 입력: GeoJSON 유지
- DB 저장: `ST_GeomFromGeoJSON(polygon_json)`
- 좌표계 지정: `ST_SetSRID(..., 4326)`
- 인덱스: `GIST (geom)`

이렇게 두면 프런트 계약은 거의 바꾸지 않고 DB 계층만 PostGIS로 확장할 수 있다.

## 8. 핵심 사용자 흐름

### 시나리오 1. 폴리곤 생성

1. 사용자가 도면 화면에 진입한다.
2. 사용자가 영역 그리기 버튼을 선택한다.
3. 도면 위에 클릭하여 다각형을 작성한다.
4. 작성 완료 후 이름과 설명을 입력한다.
5. 저장 버튼을 클릭한다.
6. DB에 polygon을 저장한다.
7. 저장된 영역이 화면에 재표출된다.

### 시나리오 2. 폴리곤 조회

1. 사용자가 도면 화면에 진입한다.
2. 해당 도면의 저장된 polygon 목록을 조회한다.
3. 폴리곤이 화면에 표시된다.
4. 사용자가 특정 polygon을 클릭한다.
5. 우측 패널에 상세정보가 표시된다.

### 시나리오 3. 폴리곤 수정

1. 사용자가 기존 polygon을 선택한다.
2. 편집 모드로 진입한다.
3. 꼭짓점 또는 메타정보를 수정한다.
4. 저장 시 DB의 polygon 정보가 갱신된다.

### 시나리오 4. 폴리곤 삭제

1. 사용자가 삭제할 polygon을 선택한다.
2. 삭제 액션을 수행한다.
3. DB에서 해당 polygon을 제거하거나 비활성 처리한다.
4. 화면과 목록에서 즉시 반영된다.

## 11. 리스크와 대응

### 리스크 1. 좌표계 혼동

도면 이미지 기반 좌표와 지도 좌표계가 혼재되면 저장 및 재표출이 꼬일 수 있다.

대응:

- MVP에서는 좌표계 기준을 명확히 하나로 제한
- 도면 기반이면 이미지 픽셀 좌표, 지도 기반이면 경위도 좌표로 고정

### 리스크 2. Draw 라이브러리 의존성

MapLibre 자체만으로는 편집 기능이 부족해 라이브러리 선택이 중요하다.

대응:

- 검증된 draw/edit 라이브러리를 우선 채택
- 초기에는 polygon 생성과 수정 중심으로 범위를 제한

### 리스크 3. SQLite 확장 한계

데이터가 커지거나 공간 연산이 필요해지면 SQLite 구조는 한계가 있다.

대응:

- MVP에서는 CRUD와 단순 조회만 구현
- 운영 또는 고도화 단계에서 PostGIS 전환을 전제로 설계
- 전환 조건은 polygon 수 증가, 공간 교차 질의 빈도 증가, 동시 사용자 증가로 정의

---

## 12. 최종 정리

이번 MVP는 부가정보 시각화 서비스가 아니라 도면 또는 지도 위에서 사용자가 영역을 직접 정의하고, 그 결과를 폴리곤으로 저장 및 관리하는 기능 검증용 프로토타입으로 정의하는 것이 맞다.

따라서 기술 스택은 FastAPI, SQLite, MapLibre GL JS, polygon draw/edit 계층 중심으로 단순화하고, 저장 포맷은 GeoJSON 기반으로 맞추는 방향이 가장 적절하다.
